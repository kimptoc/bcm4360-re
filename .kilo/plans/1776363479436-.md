# Project State Review — Direction and Progress

## Objective of this review
Provide a concise assessment of where the BCM4360 reverse-engineering project currently stands, what has been proven, what remains blocked, and the recommended near-term direction.

## Current state (evidence-based)

### What is complete
- Phase 1 reconnaissance is complete: core map, firmware extraction, and brcmfmac code-path analysis are done.
- Phase 3 bring-up is complete as a technical proof: patched `brcmfmac` can identify the chip, map memory correctly, and perform reliable firmware download.
- Core hardware constraints are well characterized:
  - TCM/rambase behavior (`rambase=0`, `ramsize=0xA0000`) is established.
  - 32-bit `iowrite32` writes are required for stable TCM download (wide writes hang).
  - B-bank/BAR2 unsafe regions and safe access envelope are documented.
- Phase 4 transport research produced a meaningful protocol understanding (shared-info/olmsg/offload semantics), even though the standalone offload-heavy path was judged non-viable as a final architecture.

### What is currently in progress
- Active work has moved into deep firmware/hang forensics (documented in `RESUME_NOTES.md` as post `test.96`).
- The project has identified a precise firmware hang location chain ending at `fn 0x1624c`, a busy-wait on completion flags.
- The dominant current hypothesis is interrupt/completion non-arrival (D11 PHY completion path not progressing), causing infinite wait.

### What has been convincingly disproven
- Multiple host-crash theories from earlier iterations were tested and ruled out across many controlled runs (DMA-only hypothesis, warm-up effects, simple IRQ/AER toggles, etc.).
- “Counter freeze means idle/WFI” interpretation has been corrected; current evidence supports genuine hang behavior.
- Several previously suspected call targets were proven non-root-cause by disassembly and trace reduction.

## Progress assessment

### Overall progress
- Substantial progress: this is no longer broad exploration; the project is now at targeted root-cause analysis depth with narrowed failure point and reproducible instrumentation patterns.
- The team has moved from architecture uncertainty to concrete firmware execution-path understanding.

### Maturity by layer
- Platform bring-up maturity: High
- Driver scaffolding maturity: High
- Protocol compatibility maturity: Medium-High (BCDC/msgbuf mismatch understood)
- End-to-end WiFi functionality maturity: Low (blocked on early init/hang resolution)

## Direction assessment

### Recommended near-term direction
- Continue the current focused forensic path (Phase 5-style debug harness + binary-assisted analysis) rather than reopening broad architectural options now.
- Treat the immediate goal as “resolve/characterize `fn 0x1624c` completion deadlock conditions” before any higher-level transport reintegration decisions.
- Keep using low-read-count, crash-safe probes to validate wait-struct fields and interrupt/completion state transitions at tightly defined timestamps.

### Why this direction is correct now
- The project already has enough broad protocol context; the blocker is no longer conceptual but execution-state specific.
- The currently active hypothesis is testable with minimal-risk instrumentation and has clear accept/reject criteria.
- Prematurely shifting back to large architecture work would increase uncertainty without removing the active blocker.

## Practical next checkpoints
1. Confirm wait-struct field evolution around the identified loop (`field20/field24/field28`) at deterministic time slices.
2. Correlate D11/PHY init path with expected completion signaling mechanism and verify whether ISR routing/enablement occurred.
3. If completion never signals, isolate earliest prerequisite missing state (clock/power/interrupt-mask/core-state), not just terminal hang site.
4. Only after this is pinned, revisit integration strategy (extended brcmfmac path vs standalone transport components) using new evidence.

## Executive comment
Project direction is good and progress is strong: the work has transitioned from exploratory reverse engineering to precise failure localization with actionable hypotheses. The right move is to keep pressing on the current tight debug loop until the completion-path deadlock is either resolved or formally bounded, then decide integration strategy from that result.
